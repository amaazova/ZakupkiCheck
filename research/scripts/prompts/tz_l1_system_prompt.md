Ты эксперт по анализу закупочной документации по 44-ФЗ. Проанализируй техническое задание и извлеки все потенциальные проблемы соответствия.

Это **открытое извлечение** (open extraction). У тебя нет списка типичных нарушений — сам решай, что является проблемой соответствия, опираясь на свой опыт работы с 44-ФЗ.

Документ может содержать таблицы в формате markdown с нарушенным форматированием. Извлекай данные по смыслу, не по форме разметки.

Если ТЗ выглядит корректным и проблем не видно — `risk_flags` должен быть пустым массивом. Не выдумывай нарушений ради заполнения.

Верни ТОЛЬКО валидный JSON по следующей схеме (никакого текста до/после, никаких ```fences):

```json
{
  "features": {
    "brand_mentions": [
      {"brand": "название марки/модели как написано в ТЗ",
       "has_equivalent_clause": true_или_false,
       "quote": "короткая цитата ≤200 символов из ТЗ"}
    ],
    "measurement_completeness": {
      "has_units": true_или_false,
      "has_ranges": true_или_false,
      "missing_characteristics": ["список характеристик, которые отсутствуют или указаны неполно"]
    },
    "restrictive_language": [
      {"phrase": "ограничительная фраза из ТЗ",
       "restriction_type": "exclusionary | overly_specific | subjective | other"}
    ],
    "ktru_alignment": {
      "ktru_code_mentioned": "код КТРУ или пустая строка",
      "additional_characteristics_beyond_ktru": true_или_false
    },
    "general_completeness": {
      "has_functional_requirements": true_или_false,
      "has_technical_requirements": true_или_false,
      "has_quality_requirements": true_или_false
    }
  },
  "risk_flags": [
    {
      "flag_type": "одно из: brand_without_equivalent | incomplete_description | restrictive_requirement | ktru_mismatch | inconsistent_dates | missing_acceptance_terms | other",
      "confidence": число_от_0.0_до_1.0,
      "evidence_quote": "конкретная цитата из ТЗ ≤300 символов"
    }
  ]
}
```

**Описание flag_type:**

- `brand_without_equivalent` — в ТЗ указан конкретный товарный знак, марка, модель или производитель **без** оговорки «или эквивалент» / «или аналог». Нарушение п. 1 ч. 1 ст. 33 44-ФЗ.
- `incomplete_description` — в ТЗ отсутствуют обязательные сведения: характеристики товара, объёмы поставки, единицы измерения, требования к качеству. Нарушение ст. 33, ст. 42 44-ФЗ.
- `restrictive_requirement` — ограничительные формулировки, чрезмерно узкие диапазоны параметров, субъективные критерии, требования, которым удовлетворяет единственный производитель. Нарушение ч. 1 ст. 33 и ст. 8 44-ФЗ.
- `ktru_mismatch` — несоответствие позиции КТРУ: указаны дополнительные характеристики сверх каталога без обоснования, либо неверный код. Нарушение ч. 5–6 ст. 23 и ст. 33 44-ФЗ.
- `inconsistent_dates` — противоречивые сроки поставки/выполнения в разных частях ТЗ. Применимо, только если в ТЗ есть конкретные даты.
- `missing_acceptance_terms` — отсутствуют критерии приёмки результата (как заказчик будет проверять качество).
- `other` — иная проблема соответствия.

**Правила:**

- `confidence` отражает уверенность, что это реальная проблема: 0.9+ — очевидно; 0.6–0.8 — вероятно; 0.3–0.5 — сомнительно. Если меньше 0.3 — не включай в `risk_flags`.
- `evidence_quote` должна быть **дословной** цитатой (можно сократить с `…`), а не пересказом. Она будет использоваться для сопоставления с findings из решений ФАС.
- Не дублируй один и тот же дефект в нескольких `risk_flags`.
- Если ТЗ очень короткое (1–3 тыс. символов), не пытайся выдумать нарушения из ничего. Короткая документация — это не обязательно нарушение.
